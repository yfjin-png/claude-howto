# チェックポイントと巻き戻し

チェックポイントを使用すれば、会話状態を保存し、Claude Codeセッション内の前のポイントに巻き戻すことができます。これは異なるアプローチの探索、ミスからの回復、または代替ソリューションの比較に非常に貴重です。

## 概要

チェックポイントを使用すれば、前のポイントに会話状態を保存して巻き戻すことができ、安全な実験と複数のアプローチの探索が可能です。これらは会話状態のスナップショットであり、次を含みます：
- 交換されたすべてのメッセージ
- 行われたファイル変更
- ツール使用履歴
- セッションコンテキスト

チェックポイントは異なるアプローチを探索する場合、ミスから回復する場合、または代替ソリューションを比較する場合に非常に貴重です。

## キー概念

| 概念 | 説明 |
|------|------|
| **チェックポイント** | メッセージ、ファイル、コンテキストを含む会話状態のスナップショット |
| **巻き戻し** | 前のチェックポイントに戻る、その後の変更を破棄 |
| **分岐ポイント** | 複数のアプローチが探索される前のチェックポイント |

## チェックポイントへのアクセス

チェックポイントにアクセスして管理するには、2つの主な方法があります：

### キーボードショートカットを使用
`Esc`を2回押す（`Esc` + `Esc`）してチェックポイントインターフェースを開き、保存されたチェックポイントを参照します。

### スラッシュコマンドを使用
クイックアクセス用に`/rewind`コマンド（エイリアス: `/checkpoint`）を使用します：

```bash
# 巻き戻しインターフェースを開く
/rewind

# またはエイリアスを使用
/checkpoint
```

## 巻き戻しオプション

巻き戻すと、5つのオプションのメニューが表示されます：

1. **コードと会話を復元** -- ファイルとメッセージの両方をそのチェックポイントに戻す
2. **会話を復元** -- メッセージのみを巻き戻す、現在のコードはそのままに
3. **コードを復元** -- ファイル変更のみを戻す、完全な会話履歴をそのままに
4. **ここから要約** -- このポイント以降の会話をAI生成の要約に圧縮し、コンテキストウィンドウスペースを解放。選択されたポイント前のメッセージはそのまま。ディスク上のファイルは変更されません。元のメッセージはセッション記録に保持されます。オプションで要約を特定のトピックに焦点を当てるための命令を提供できます。
5. **キャンセル** -- キャンセルして現在の状態に戻る

> **注**: 会話を復元するか要約した後、選択されたメッセージからの元のプロンプトが入力フィールドに復元されるため、再送信または編集できます。

## 自動チェックポイント

Claude Codeは自動的にチェックポイントを作成します：

- **すべてのユーザープロンプト** - 新しいチェックポイントがユーザー入力のたびに作成されます
- **永続的** - チェックポイントはセッション間で保持されます
- **自動クリーンアップ** - チェックポイントは30日後に自動的にクリーンアップされます

つまり、数分前から数日前までのセッション内の任意のポイントに常に巻き戻すことができます。

## ユースケース

| シナリオ | ワークフロー |
|---------|----------|
| **アプローチの探索** | 保存 → 試してA → 保存 → 巻き戻し → 試してB → 比較 |
| **安全なリファクタリング** | 保存 → リファクタリング → テスト → 失敗の場合: 巻き戻し |
| **A/Bテスト** | 保存 → 設計A → 保存 → 巻き戻し → 設計B → 比較 |
| **ミスク回復** | 問題に気づく → 最後の良い状態に巻き戻す |

## チェックポイント使用

### チェックポイントの表示と巻き戻し

`Esc`を2回押すか`/rewind`を使用してチェックポイントブラウザを開きます。利用可能なすべてのチェックポイントがタイムスタンプ付きでリストされます。任意のチェックポイントを選択してその状態に巻き戻します。

### チェックポイント詳細

各チェックポイントは以下を表示します：
- 作成されたタイムスタンプ
- 変更されたファイル
- 会話内のメッセージ数
- 使用されたツール

## 実践的な例

### 例1: 異なるアプローチを探索

```
ユーザー: APIにキャッシングレイヤーを追加しましょう

Claude: キャッシング層をAPIエンドポイントに追加します...
[チェックポイントAで変更を行う]

ユーザー: 実際には、代わりにインメモリキャッシングを試してみましょう

Claude: 異なるアプローチを探索するために巻き戻します...
[ユーザーがEsc+Escを押し、チェックポイントAに巻き戻す]
[チェックポイントBでインメモリキャッシングを実装]

ユーザー: これで両方のアプローチを比較できます
```

### 例2: ミスから回復

```
ユーザー: 認証モジュールをJWTを使用するようにリファクタリング

Claude: 認証モジュールをリファクタリングします...
[広範な変更を行う]

ユーザー: 待機、これはOAuth統合を壊しました。戻りましょう。

Claude: リファクタリング前の状態に巻き戻すのを助けます...
[ユーザーがEsc+Escを押し、リファクタリング前のチェックポイントを選択]

ユーザー: 今回はより保守的なアプローチを試してみましょう
```

### 例3: 安全な実験

```
ユーザー: これを関数スタイルで書き直してみましょう
[実験前にチェックポイント作成]

Claude: [実験的な変更を行う]

ユーザー: テストが失敗しています。巻き戻しましょう。
[ユーザーがEsc+Escを押してチェックポイントに巻き戻す]

Claude: 変更を巻き戻しました。異なるアプローチを試してみましょう。
```

### 例4: アプローチを分岐

```
ユーザー: 2つのデータベース設計を比較したいです
[チェックポイントを記録 - 「開始」と呼ぶ]

Claude: 最初の設計を作成します...
[スキーマAを実装]

ユーザー: 今、戻って2番目のアプローチを試してみましょう
[ユーザーがEsc+Escを押し、「開始」に巻き戻す]

Claude: これで2番目のスキーマを実装します...
[スキーマBを実装]

ユーザー: 素晴らしい！今は両方のスキーマから選べます
```

## チェックポイント保持

Claude Codeはチェックポイントを自動的に管理します：

- チェックポイントはすべてのユーザープロンプトで自動的に作成されます
- 古いチェックポイントは最大30日間保持されます
- チェックポイントは無制限のストレージ増大を防ぐために自動的にクリーンアップされます

## ワークフローパターン

### 探索の分岐戦略

複数のアプローチを探索するとき：

```
1. 初期実装を開始 → チェックポイントA
2. アプローチ1を試す → チェックポイントB
3. チェックポイントAに巻き戻す
4. アプローチ2を試す → チェックポイントC
5. BとCの結果を比較
6. 最良のアプローチを選択して続行
```

### 安全なリファクタリングパターン

重大な変更を行うとき：

```
1. 現在の状態 → チェックポイント（自動）
2. リファクタリング開始
3. テストを実行
4. テストが成功 → 続行
5. テストが失敗 → 巻き戻して異なるアプローチを試す
```

## ベストプラクティス

チェックポイントはすべてのユーザープロンプトで自動的に作成されるため、手動で保存状態について心配することなく作業に焦点を当てることができます。ただし、これらのプラクティスを念頭に置いてください：

### チェックポイントを効果的に使用

✅ **すべき:**
- 巻き戻す前に利用可能なチェックポイントを確認
- 異なる方向を探索したいときに巻き戻しを使用
- チェックポイントを保存して異なるアプローチを比較
- 各巻き戻しオプション（コードと会話を復元、会話を復元、コードを復元、または要約）が何をするかを理解

❌ **してはいけない:**
- チェックポイントのみに頼ってコードを保護
- チェックポイントが外部ファイルシステムの変更を追跡するのを期待
- チェックポイントをgitコミットの代わりとして使用

## 設定

チェックポイントはClaude Codeのデフォルト動作であり、有効にするための設定は必要ありません。すべてのユーザープロンプトは自動的にチェックポイントを作成します。

唯一のチェックポイント関連の設定は、セッションとチェックポイントが保持される期間を制御する`cleanupPeriodDays`です：

```json
{
  "cleanupPeriodDays": 30
}
```

- `cleanupPeriodDays`: セッション履歴とチェックポイントを保持する日数（デフォルト: `30`）

## 制限

チェックポイントには以下の制限があります：

- **Bashコマンド変更は追跡されません** - `rm`、`mv`、`cp`などのファイルシステム上の操作はチェックポイントで記録されません
- **外部変更は追跡されません** - Claude Codeの外部で行われた変更（エディタ、ターミナル等）は記録されません
- **バージョン管理の代替ではありません** - コードベースへの永続的で監査可能な変更にはgitを使用

## トラブルシューティング

### チェックポイント不足

**問題**: 期待されるチェックポイントが見つかりません

**解決策**:
- チェックポイントがクリアされたかをチェック
- ディスク容量をチェック
- `cleanupPeriodDays`が十分に高く設定されていることを確認（デフォルト: 30日）

### 巻き戻しに失敗

**問題**: チェックポイントに巻き戻せません

**解決策**:
- コミットされていない変更が競合していないか確認
- チェックポイントが破損していないか確認
- 異なるチェックポイントに巻き戻してみてください

## Gitとの統合

チェックポイントはgitを補足します（置き換えません）：

| 機能 | Git | チェックポイント |
|------|-----|-----------------|
| スコープ | ファイルシステム | 会話+ファイル |
| 永続性 | 永続 | セッションベース |
| 粒度 | コミット | 任意のポイント |
| スピード | 遅い | 即座 |
| 共有 | はい | 限定的 |

両方を一緒に使用：
1. 高速実験にはチェックポイントを使用
2. 最終化された変更にはgitコミットを使用
3. git操作の前にチェックポイントを作成
4. チェックポイント状態をgitにコミット

## クイックスタートガイド

### 基本ワークフロー

1. **通常通り作業** - Claude Codeは自動的にチェックポイントを作成
2. **戻りたい？** - `Esc`を2回押すか`/rewind`を使用
3. **チェックポイントを選択** - リストから選択して巻き戻す
4. **復元内容を選択** - コードと会話を復元、会話を復元、コードを復元、ここから要約、またはキャンセル
5. **作業を続行** - そのポイントに戻りました

### キーボードショートカット

- **`Esc` + `Esc`** - チェックポイントブラウザを開く
- **`/rewind`** - チェックポイントにアクセスする別の方法
- **`/checkpoint`** - `/rewind`のエイリアス

## コンテキストモニタリング：巻き戻すべき時を知る

チェックポイントで戻ることができます — しかし*いつ*戻るべきかをどのように知ることができますか？会話が成長するにつれて、Claudeのコンテキストウィンドウが満杯になり、モデルの品質は静かに低下します。気付かずに半盲のモデルからコードをシップしているかもしれません。

**[cc-context-stats](https://github.com/luongnv89/cc-context-stats)**はリアルタイムの**コンテキストゾーン**をClaude Codeステータスバーに追加することでこれを解決します。計画（緑、安全に計画とコード）からコード（黄色、新しい計画を開始しないようにしましょう）からダンプ（オレンジ、完了して巻き戻す）まで、コンテキストウィンドウ内のどこにいるかを追跡します。ゾーンが移動したら、劣化した出力で推し進めるのではなく、チェックポイントを作成して新しく始める時間です。

## 概要

チェックポイントはClaude Codeの自動機能で、仕事の喪失を恐れることなく異なるアプローチを安全に探索することができます。すべてのユーザープロンプトは新しいチェックポイントを自動的に作成するため、セッション内の任意の前のポイントに巻き戻すことができます。

主な利点：
- 複数のアプローチを恐れることなく実験
- ミスから迅速に回復
- 異なるソリューションを並べて比較
- バージョン管理システムと安全に統合

覚えていてください: チェックポイントはgitの代替ではありません。チェックポイントを高速実験に使用し、gitを永続的なコード変更に使用します。
