# プランニングモード例

Claude Codeでのプランニングモードの効果的な使用を示す実世界の例。

## 例1: REST APIを構築

### プランニングモードなし

```
ユーザー: ブログ用のREST APIを構築

Claude: APIを作成します...
[すぐにコーディング開始、要件を見落とす可能性]
```

### プランニングモード付き

```
ユーザー: /plan ブログ用のREST APIを構築

Claude: ブログAPIの包括的なプランを作成しましょう。

## ブログREST API実装計画

### フェーズ1: プロジェクトセットアップ（15分）
1. Node.jsプロジェクトをExpressで初期化
2. TypeScript設定をセットアップ
3. 依存性をインストール（express、pg、jwt、joi）
4. 環境変数を設定
5. プロジェクト構造をセットアップ

### フェーズ2: データベースレイヤー（30分）
6. データベーススキーマを設計
   - usersテーブル（id、email、password_hash、created_at）
   - postsテーブル（id、user_id、title、content、published、created_at、updated_at）
   - commentsテーブル（id、post_id、user_id、content、created_at）
7. マイグレーションファイル作成
8. データベース接続プールをセットアップ
9. データベースモデルを実装

### フェーズ3: 認証（45分）
10. ユーザー登録エンドポイント実装
11. JWTでログインエンドポイント実装
12. 認証ミドルウェア作成
13. bcryptでパスワードハッシング追加
14. トークン更新ロジック実装

### フェーズ4: Posts API（40分）
15. GET /api/posts - すべてのパブリッシュ済みpostsをリスト（ページネーション付き）
16. GET /api/posts/:id - 単一postを取得
17. POST /api/posts - 新しいpostを作成（認証必須）
18. PUT /api/posts/:id - postを更新（認証必須、所有権チェック）
19. DELETE /api/posts/:id - postを削除（認証必須、所有権チェック）
20. GET /api/users/:id/posts - ユーザーのpostsを取得

### フェーズ5: Comments API（30分）
21. GET /api/posts/:id/comments - postコメント取得
22. POST /api/posts/:id/comments - コメント追加（認証必須）
23. DELETE /api/comments/:id - コメント削除（認証必須、所有権チェック）

### フェーズ6: 検証とエラーハンドリング（30分）
24. Joiでリクエスト検証追加
25. グローバルエラーハンドラ実装
26. 入力のサニタイズを追加
27. レート制限を実装

### フェーズ7: テスト（60分）
28. モデル用のユニットテスト書き込み
29. 認証エンドポイント用の統合テスト書き込み
30. postsエンドポイント用の統合テスト書き込み
31. commentsエンドポイント用の統合テスト書き込み
32. 80%+コードカバレッジを達成

### フェーズ8: ドキュメンテーションとデプロイ（30分）
33. SwaggerでAPIドキュメンテーション生成
34. セットアップ指示付きREADME作成
35. Dockerの設定を追加
36. CI/CDパイプラインをセットアップ

**推定時間**: 4.5時間
**作成するファイル**: ~25
**主要技術**: Express、PostgreSQL、JWT、TypeScript

**APIエンドポイント概要**:
- 認証: 2エンドポイント
- Posts: 6エンドポイント
- Comments: 3エンドポイント
- 合計: 11エンドポイント

準備ができていますか？（yes/no/modify）

ユーザー: yes

Claude: フェーズ1を開始: プロジェクトセットアップ...
[各フェーズを体系的に実装]
```

## 例2: データベース移行

### プランニングモード付き

```
ユーザー: /plan MongoDBからPostgreSQLへの移行

Claude: これは複雑な移行です。詳細な計画を作成しましょう。

## MongoDBからPostgreSQL移行計画

### フェーズ1: 分析と準備（1日目）
1. 現在のMongoDBスキーマを分析
2. データ関係と制約を特定
3. 現在のデータボリュームと成長レートを文書化
4. 同等なPostgreSQLスキーマを設計
5. データ型マッピングを計画
6. 潜在的な移行チャレンジを特定

### フェーズ2: PostgreSQLセットアップ（1日目）
7. PostgreSQLデータベースをセットアップ
8. テーブル、インデックス、制約付きスキーマ作成
9. コネクション プール設定
10. バックアップ戦略を設定

### フェーズ3: デュアルライト実装（2-3日目）
11. データベース抽象化レイヤー作成
12. 両方のデータベースへのデュアルライト実装
13. データ一貫性チェック追加
14. ステージングにデュアルライトをデプロイ
15. 問題について監視

### フェーズ4: データ移行（4日目）
16. MongoDBのデータエクスポートスクリプト作成
17. PostgreSQLのデータインポートスクリプト作成
18. データ変換ロジック実装
19. 本番データのコピーで初期移行実行
20. 移行されたデータの整合性を検証

### フェーズ5: 段階的カットオーバー（5-7日目）
21. 本番でデュアルライトを有効化
22. 過去データをバッチで移行
23. 継続的にデータ一貫性を検証
24. 両方のデータベースを監視
25. データ同期化の問題を修正

### フェーズ6: PostgreSQL主要（8日目）
26. 読み取り操作をPostgreSQLに切り替え
27. クエリパフォーマンスを監視
28. 遅いクエリを最適化
29. セーフティのためデュアルライト継続

### フェーズ7: MongoDB廃止（9-10日目）
30. MongoDBへの書き込みを停止
31. 安全性のためMongoDBをバックアップとして保持（2週間）
32. コードからMongoDBの依存性を削除
33. MongoDBデータをアーカイブ
34. MongoDBインスタンスをデコミッション

**推定時間**: 10日
**リスクレベル**: 高
**ロールバック戦略**: デュアルライトでMongoDBを実行

準備しますか？（yes/no/modify）

ユーザー: Modify - 各フェーズ用のロールバック計画を追加しましょう

Claude: 各フェーズ用のロールバック手順付き更新されたプラン...
```

## 例3: フロントエンドリファクタリング

```
ユーザー: /plan Reactアプリをクラスコンポーネントからフックにリファクタリング

Claude: 系統的なリファクタリング計画を作成します。

## Reactクラスからフック移行計画

### フェーズ1: 準備と分析（2時間）
1. すべてのクラスコンポーネントを監査（すべて.jsx/.tsxファイル）
2. 複雑さで分類：
   - シンプル（状態のみ）: ~15コンポーネント
   - 中程度（ライフサイクルメソッド）: ~23コンポーネント
   - 複雑（複数ライフサイクル、refs）: ~8コンポーネント
3. 更新が必要なサードパーティライブラリを特定
4. コンポーネント依存性グラフを作成
5. 移行順序を計画（葉ノードから）

### フェーズ2: セットアップとツーリング（1時間）
6. Reactをv18+に更新
7. フック用のESLintルールをセットアップ
8. 変換テンプレート作成
9. 自動テストをセットアップ
10. 開始前にチェックポイント作成

### フェーズ3: シンプルなコンポーネント移行（4時間）
11. シンプルなステートフルコンポーネント（15コンポーネント）を変換
12. this.stateをuseStateで置換
13. this.setStateを状態セッターで置換
14. 変換後に各コンポーネントをテスト
15. テストを更新

### フェーズ4: 中程度のコンポーネント移行（8時間）
16. ライフサイクルメソッド付きコンポーネント（23コンポーネント）を変換
17. componentDidMountをuseEffectで置換
18. componentDidUpdateをuseEffectで置換
19. componentWillUnmountをuseEffect クリーンアップで置換
20. 徹底的にテスト

### フェーズ5: 複雑なコンポーネント移行（12時間）
21. 複雑なコンポーネント（8コンポーネント）を変換
22. refをuseRefで置換
23. 共有ロジック用にカスタムフック抽出
24. エッジケースを処理（複数のuseEffect）
25. 拡張テスト

### フェーズ6: 共有ロジック抽出（6時間）
26. コンポーネント間の重複ロジック特定
27. カスタムフック作成：
    - useAuth
    - useFetch
    - useLocalStorage
    - useDebounce
28. カスタムフックを使用するようコンポーネントをリファクタ
29. カスタムフック独立テスト

### フェーズ7: パフォーマンス最適化（4時間）
30. 必要に応じてReact.memoを追加
31. useCallbackで最適化
32. useMemoで最適化
33. パフォーマンスを中心にして回帰修正

### フェーズ8: クリーンアップとドキュメンテーション（2時間）
34. 未使用のクラスコンポーネントユーティリティを削除
35. ドキュメンテーションを更新
36. コーディングガイドラインを更新
37. チーム向けの移行ガイド作成

**推定時間**: 5日（39時間）
**変更するファイル**: 46コンポーネント
**リスクレベル**: 中程度

**移行順序**:
1. リーフコンポーネント（依存なし）
2. 中間レベルコンポーネント
3. トップレベルコンテナコンポーネント
4. App.jsx（最後）

**テスト戦略**:
- 各コンポーネント用のユニットテスト
- ユーザーフロー用の統合テスト
- ビジュアル回帰テスト
- パフォーマンスベンチマーク

**成功メトリクス**:
- すべてのテストが成功
- ビジュアル回帰なし
- パフォーマンス保持または改善
- バンドルサイズが大幅に増加しない

進める？（yes/no/modify）
```

## キーテイクアウェイ

### プランニングモードの利点

1. **明確性**: コーディング開始前のロードマップ
2. **推定**: 時間と努力の推定
3. **リスク評価**: 実装前の潜在的な問題を特定
4. **優先順位**: タスクの論理的な順序
5. **承認**: 実行前にレビューと承認
6. **変更**: フィードバックに基づいてプランを調整

### プランニングモードを使用すべき時

✅ **常に使用**:
- マルチ日プロジェクト
- チーム協力
- 重大なシステム変更
- 新しい概念学習
- 複雑なリファクタリング

❌ **使用しないでください**:
- バグ修正
- 小さなツイーク
- シンプルなクエリ
- クイック実験

### ベストプラクティス

1. **計画を注意深くレビュー** 承認前
2. **計画を変更** 問題を見つけたら
3. **複雑なタスクを分解** してください
4. **現実的な推定** 時間枠
5. **ロールバック戦略を含める** で
6. **成功基準を追加** してください
7. **各フェーズでテストを計画** してください
