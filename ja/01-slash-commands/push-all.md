# すべてコミットしてプッシュ

⚠️ **警告**: すべての変更をステージし、リモートにコミットしてプッシュします。すべての変更が一緒に属していると確信している場合のみ使用してください。

## ワークフロー

### 1. 変更を分析

並列実行：
- `git status` - 変更/追加/削除/未追跡ファイルを表示
- `git diff --stat` - 変更統計を表示
- `git log -1 --oneline` - メッセージスタイル用の最近のコミットを表示

### 2. 安全性チェック

**❌ 検出された場合は停止して警告:**
- シークレット: `.env*`、`*.key`、`*.pem`、`credentials.json`、`secrets.yaml`、`id_rsa`、`*.p12`、`*.pfx`、`*.cer`
- API キー: 実際の値を持つ `*_API_KEY`、`*_SECRET`、`*_TOKEN` 変数 (`your-api-key`、`xxx`、`placeholder` のようなプレースホルダーではない)
- 大容量ファイル: `>10MB` (Git LFS なし)
- ビルドアーティファクト: `node_modules/`、`dist/`、`build/`、`__pycache__/`、`*.pyc`、`.venv/`
- 一時ファイル: `.DS_Store`、`thumbs.db`、`*.swp`、`*.tmp`

**API キー検証:**
次のようなパターンの修正ファイルをチェック：
```bash
OPENAI_API_KEY=sk-proj-xxxxx  # ❌ 実際のキーが検出！
AWS_SECRET_KEY=AKIA...         # ❌ 実際のキーが検出！
STRIPE_API_KEY=sk_live_...    # ❌ 実際のキーが検出！

# ✅ 許可可能なプレースホルダー：
API_KEY=your-api-key-here
SECRET_KEY=placeholder
TOKEN=xxx
API_KEY=<your-key>
SECRET=${YOUR_SECRET}
```

**✅ 検証:**
- `.gitignore` が適切に構成されている
- マージコンフリクトがない
- 正しいブランチ (main/master の場合は警告)
- API キーはプレースホルダーのみ

### 3. 確認をリクエスト

概要を提示：
```
📊 変更概要：
- X個のファイルが修正、Y個が追加、Z個が削除
- 合計: +AAA 挿入、-BBB 削除

🔒 安全性: ✅ シークレットなし | ✅ 大容量ファイルなし | ⚠️ [警告]
🌿 ブランチ: [name] → origin/[name]

実行内容: git add . → コミット → プッシュ

続行するには 'yes' を入力、キャンセルするには 'no' を入力してください。
```

**続行前に明示的に「yes」を待機してください。**

### 4. 実行 (確認後)

順序で実行：
```bash
git add .
git status  # ステージング検証
```

### 5. コミットメッセージを生成

変更を分析し、従来のコミットを作成：

**形式:**
```
[type]: 簡潔な要約 (最大72文字)

- 重要な変更1
- 重要な変更2
- 重要な変更3
```

**タイプ:** `feat`、`fix`、`docs`、`style`、`refactor`、`test`、`chore`、`perf`、`build`、`ci`

**例:**
```
docs: コンセプト README ファイルを包括的ドキュメントで更新

- アーキテクチャダイアグラムとテーブルを追加
- 実践例を含める
- ベストプラクティスセクションを拡張
```

### 6. コミットしてプッシュ

```bash
git commit -m "$(cat <<'EOF'
[生成されたコミットメッセージ]
EOF
)"
git push  # 失敗する場合: git pull --rebase && git push
git log -1 --oneline --decorate  # 検証
```

### 7. 成功を確認

```
✅ リモートへのプッシュに成功しました！

コミット: [hash] [message]
ブランチ: [branch] → origin/[branch]
ファイル変更: X (+挿入、-削除)
```

## エラーハンドリング

- **git add が失敗**: パーミッションをチェック、ロックされたファイル、リポジトリが初期化されていることを確認
- **git commit が失敗**: プリコミットフックを修正、git config (user.name/email) をチェック
- **git push が失敗**:
  - ファストフォワード不可: `git pull --rebase && git push`
  - リモートブランチがない: `git push -u origin [branch]`
  - 保護されたブランチ: 代わりに PR ワークフローを使用

## 使用時期

✅ **良い：**
- 複数ファイルのドキュメント更新
- テストとドキュメント付き機能
- ファイル全体のバグ修正
- プロジェクト全体のフォーマット/リファクタリング
- 構成変更

❌ **避ける：**
- コミット対象が不確実
- シークレット/機密データを含む
- プリコミットフックが失敗している
- マージコンフリクトが存在
- マージコンフリクトが存在する場合
- 粒度の高いコミット履歴が必要
- プリコミットフック失敗

## 代替案

ユーザーがコントロールを希望する場合は、以下を提案：
1. **選別ステージング**: 特定のファイルをレビュー/ステージ
2. **対話的ステージング**: パッチ選択用の `git add -p`
3. **PR ワークフロー**: ブランチ作成 → プッシュ → PR (スラッシュコマンド `/pr` を使用)

**⚠️ 注意**: プッシュ前に常に変更をレビューしてください。不確実な場合、より細かく制御するために個別の git コマンドを使用してください。
