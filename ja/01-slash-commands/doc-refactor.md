# ドキュメント再構成

プロジェクトタイプに適応させたプロジェクトドキュメント構造をリファクタリングします：

1. **プロジェクトを分析**: タイプ (ライブラリ/API/ウェブアプリ/CLI/マイクロサービス)、アーキテクチャ、ユーザーペルソナを特定
2. **ドキュメントを一元化**: 技術ドキュメントを `docs/` に移動し、適切なクロスリファレンスを実施
3. **ルート README.md**: エントリポイントとして、概要、クイックスタート、モジュール/コンポーネント概要、ライセンス、連絡先でストリームライン化
4. **コンポーネント ドキュメント**: モジュール/パッケージ/サービスレベルの README ファイルをセットアップと テスト指示付きで追加
5. **`docs/` を整理**: 関連カテゴリ別に：
   - アーキテクチャ、API リファレンス、データベース、デザイン、トラブルシューティング、デプロイ、コントリビューション (プロジェクト需要に適応)
6. **ガイドを作成** (該当するものを選択)：
   - ユーザーガイド: アプリケーション用のエンドユーザードキュメント
   - API ドキュメント: API 用のエンドポイント、認証、例
   - 開発ガイド: セットアップ、テスト、コントリビューション ワークフロー
   - デプロイメント ガイド: サービス/アプリ用の本番環境デプロイ
7. **すべてのダイアグラムに Mermaid を使用** (アーキテクチャ、フロー、スキーマ)

ドキュメントを簡潔、スキャン可能、およびプロジェクトタイプに応じた文脈で保つようにしてください。
