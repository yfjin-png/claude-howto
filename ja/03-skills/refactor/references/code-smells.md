# コード スメル カタログ

Martin Fowlerの *Refactoring*（第2版）に基づいた、包括的なコード スメルのリファレンス。コード スメルはシステムのより深い問題に対応する表面的な兆候です。

> 「コード スメルは、通常、システムのより深い問題に対応する表面的な指標です。」— Martin Fowler

---

## ブローター

効果的に処理するには大きくなりすぎたものを表すコード スメル。

### Long Method

**サイン：**
- メソッドが30～50行を超える
- メソッド全体を表示するためにスクロール必要
- ネストの複数レベル
- セクションが何をするかを説明するコメント

**なぜ悪いのか：**
- 理解しづらい
- 分離でテストするのが難しい
- 変更は意図しない結果を生む
- 重複ロジックが隠れている

**リファクタリング：**
- メソッドを抽出
- Tempをクエリで置き換える
- パラメーター オブジェクトを導入
- メソッドをメソッド オブジェクトで置き換える
- 条件を分解

**例（前）：**
```javascript
function processOrder(order) {
  // 注文の検証（20行）
  if (!order.items) throw new Error('アイテムなし');
  if (order.items.length === 0) throw new Error('空の注文');
  // ...さらに検証

  // 合計を計算（30行）
  let subtotal = 0;
  for (const item of order.items) {
    subtotal += item.price * item.quantity;
  }
  // ...税、送料、割引

  // 通知を送信（20行）
  // ...メール ロジック
}
```

**例（後）：**
```javascript
function processOrder(order) {
  validateOrder(order);
  const totals = calculateOrderTotals(order);
  sendOrderNotifications(order, totals);
  return { order, totals };
}
```

### Large Class

**サイン：**
- クラスには多くのインスタンス変数がある（>7-10）
- クラスには多くのメソッドがある（>15-20）
- クラス名が曖昧（マネージャー、ハンドラー、プロセッサー）
- メソッドはすべてのインスタンス変数を使用しない

**なぜ悪いのか：**
- 単一責任原則に違反
- テストするのが難しい
- 変更は関連のない機能に及ぶ
- パーツを再利用するのが難しい

**リファクタリング：**
- クラスを抽出
- サブクラスを抽出
- インターフェイスを抽出

**検出：**
```
コード行 > 300
メソッド数 > 15
フィールド数 > 10
```

### Primitive Obsession

**サイン：**
- ドメイン概念（メール用の文字列、金銭用の整数）にプリミティブを使用
- オブジェクトの代わりにプリミティブ配列
- 型コード用の文字列定数
- マジック ナンバー/文字列

**なぜ悪いのか：**
- タイプ レベルでの検証なし
- ロジックがコードベース全体に散在
- 間違った値を渡すのが簡単
- ドメイン概念が欠落している

**リファクタリング：**
- プリミティブをオブジェクトで置き換える
- 型コードをクラスで置き換える
- 型コードをサブクラスで置き換える
- 型コードを状態/戦略で置き換える

**例（前）：**
```javascript
const user = {
  email: 'john@example.com',     // ただの文字列
  phone: '1234567890',           // ただの文字列
  status: 'active',              // マジック文字列
  balance: 10050                 // センティメートル整数
};
```

**例（後）：**
```javascript
const user = {
  email: new Email('john@example.com'),
  phone: new PhoneNumber('1234567890'),
  status: UserStatus.ACTIVE,
  balance: Money.cents(10050)
};
```

### Long Parameter List

**サイン：**
- メソッドに4以上のパラメーター
- 常に一緒に表示されるパラメーター
- メソッド動作を変更するブール フラグ
- 頻繁に渡されるnull/undefined

**なぜ悪いのか：**
- 正しく呼び出すのが難しい
- パラメーター順序の混乱
- メソッドが多すぎる場合を示す
- 新しいパラメーターを追加するのが難しい

**リファクタリング：**
- パラメーター オブジェクトを導入
- オブジェクト全体を保持
- パラメーターをメソッド呼び出しで置き換える
- フラグ引数を削除

**例（前）：**
```javascript
function createUser(firstName, lastName, email, phone,
                    street, city, state, zip,
                    isAdmin, isActive, createdBy) {
  // ...
}
```

**例（後）：**
```javascript
function createUser(personalInfo, address, options) {
  // personalInfo: { firstName, lastName, email, phone }
  // address: { street, city, state, zip }
  // options: { isAdmin, isActive, createdBy }
}
```

---

### Data Clumps

**サイン：**
- 同じ3個以上のフィールドが繰り返し表示
- 常に一緒に移動するパラメーター
- フィールド サブセットが一緒に属するクラス

**なぜ悪いのか：**
- 処理ロジックが重複
- 抽象化が不足している
- 拡張するのが難しい
- 隠されたクラスを示す

**リファクタリング：**
- クラスを抽出
- パラメーター オブジェクトを導入
- オブジェクト全体を保持

**例：**
```javascript
// データ クランプ：（x、y、z）座標
function movePoint(x, y, z, dx, dy, dz) { }
function scalePoint(x, y, z, factor) { }
function distanceBetween(x1, y1, z1, x2, y2, z2) { }

// Point3Dクラスを抽出
class Point3D {
  constructor(x, y, z) { }
  move(delta) { }
  scale(factor) { }
  distanceTo(other) { }
}
```

## オブジェクト指向の乱用

OOP原則の不完全または不正な使用を示すスメル。

### Switch Statements

**サイン：**
- 長いswitch/case またはif/elseチェーン
- 複数の場所で同じswitch
- 型コードのスイッチ
- 新しいケースを追加するには、すべてをスイッチする必要があります

**なぜ悪いのか：**
- Open/Closed原則に違反
- すべてのスイッチ場所への変更リップル
- 拡張するのが難しい
- 多くの場合、多態性が欠落している

**リファクタリング：**
- 条件を多態性で置き換える
- 型コードをサブクラスで置き換える
- 型コードを状態/戦略で置き換える

**例（前）：**
```javascript
function calculatePay(employee) {
  switch (employee.type) {
    case 'hourly':
      return employee.hours * employee.rate;
    case 'salaried':
      return employee.salary / 12;
    case 'commissioned':
      return employee.sales * employee.commission;
  }
}
```

**例（後）：**
```javascript
class HourlyEmployee {
  calculatePay() {
    return this.hours * this.rate;
  }
}

class SalariedEmployee {
  calculatePay() {
    return this.salary / 12;
  }
}
```

### Temporary Field

**サイン：**
- インスタンス変数は一部のメソッドでのみ使用
- フィールドは条件付きで設定
- 特定のケースの複雑な初期化

**なぜ悪いのか：**
- 混乱させる — フィールドは存在しますが、nullの場合があります
- オブジェクト状態を理解するのが難しい
- 条件ロジックが隠れていることを示す

**リファクタリング：**
- クラスを抽出
- Nullオブジェクトを導入
- Tempフィールドをローカルで置き換える

---

### Refused Bequest

**サイン：**
- サブクラスは継承メソッド/データを使用しない
- サブクラスは何もしないとオーバーライド
- コード再利用にはIS-A関係ではなく継承を使用

**なぜ悪いのか：**
- 間違った抽象化
- Liskov置換原則に違反
- 誤解を招く階層

**リファクタリング：**
- メソッド/フィールドをプッシュ ダウン
- サブクラスをデリゲートで置き換える
- 継承をデリゲーションで置き換える

---

### Alternative Classes with Different Interfaces

**サイン：**
- 同様のことを行う2つのクラス
- 同じ概念に対して異なるメソッド名
- 相互に使用できる

**なぜ悪いのか：**
- 実装の重複
- 共通インターフェイスなし
- 切り替えるのが難しい

**リファクタリング：**
- メソッドの名前を変更
- メソッドを移動
- スーパークラスを抽出
- インターフェイスを抽出

## 変更防止

変更を困難にするスメル — 1つを変更するには、多くの変更が必要です。

### Divergent Change

**サイン：**
- 1つのクラスが複数の理由で変更
- 異なる領域の変更が同じクラスの編集をトリガー
- クラスは「神クラス」

**なぜ悪いのか：**
- 単一責任に違反
- 変更頻度が高い
- マージの競合

**リファクタリング：**
- クラスを抽出
- スーパークラスを抽出
- サブクラスを抽出

**例：**
ユーザークラスは以下の変更：
- 認証の変更
- プロフィール変更
- 請求変更
- 通知の変更

→ 抽出：`AuthService`、`ProfileService`、`BillingService`、`NotificationService`

---

### Shotgun Surgery

**サイン：**
- 1つの変更には多くのクラスでの編集が必要
- 小さな機能は10以上のファイルにタッチ
- 変更は散らばっており、すべてを見つけるのが難しい

**なぜ悪いのか：**
- 簡単に見逃す
- 高いカップリング
- 変更はエラーが起こりやすい

**リファクタリング：**
- メソッドを移動
- フィールドを移動
- クラスをインライン化

**検出：**
1つのフィールドを追加するには、5個以上のファイルで変更が必要です。

---

### Parallel Inheritance Hierarchies

**サイン：**
- 1つの階層でサブクラスを作成するには、別の階層でサブクラスが必要
- クラス プレフィックスが一致（例：`DatabaseOrder`、`DatabaseProduct`）

**なぜ悪いのか：**
- メンテナンスが2倍
- 階層間のカップリング
- 片側を忘れるのが簡単

**リファクタリング：**
- メソッドを移動
- フィールドを移動
- 1つの階層を削除

## 廃棄物

不要で削除する必要があるもの。

### Comments（過度）

**サイン：**
- コードが何をするかを説明するコメント
- コメント化されたコード
- 悩みのままのTODO/FIXME
- コメントでの謝罪

**なぜ悪いのか：**
- コメント嘘つき（同期されません）
- コードは自己説明的である必要があります
- デッド コードは混乱を引き起こす

**リファクタリング：**
- メソッドを抽出（名前が説明する）
- 名前変更（コメントなしで明確）
- コメント化されたコードを削除
- アサーションを導入

**良い対悪いコメント：**
```javascript
// 悪い：何を説明するか
// ユーザーをループしてアクティブであるかチェック
for (const user of users) {
  if (user.status === 'active') { }
}

// 良い：なぜを説明するか
// アクティブなユーザーのみ - 非アクティブはクリーンアップ ジョブで処理
const activeUsers = users.filter(u => u.isActive);
```

---

### Duplicate Code

**サイン：**
- 複数の場所で同じコード
- 小さなバリエーションを含む同様のコード
- コピーしてペイストする パターン

**なぜ悪いのか：**
- 複数の場所でのバグ修正が必要
- 矛盾リスク
- ブロートウェア コードベース

**リファクタリング：**
- メソッドを抽出
- クラスを抽出
- メソッドをプルアップ（階層）
- テンプレート メソッドを形成

**検出ルール：**
3回以上複製されたコードを抽出する必要があります。

---

### Lazy Class

**サイン：**
- クラスは存在を正当化するには不十分
- 追加値のないラッパー
- 過度設計の結果

**なぜ悪いのか：**
- メンテナンス オーバーヘッド
- 不要な間接参照
- 利益なしで複雑さ

**リファクタリング：**
- クラスをインライン化
- 階層を折りたたむ

---

### Dead Code

**サイン：**
- 到達不可能なコード
- 未使用の変数/メソッド/クラス
- コメント化されたコード
- 不可能な条件の背後のコード

**なぜ悪いのか：**
- 混乱
- メンテナンス上の負担
- 理解を遅くする

**リファクタリング：**
- デッド コードを削除
- セーフ削除

**検出：**
```bash
# 未使用のエクスポートを探す
# 参照されていない関数を探す
# IDE「未使用」の警告
```

---

### Speculative Generality

**サイン：**
- 1つのサブクラスを持つ抽象クラス
- 「将来の使用」用の未使用パラメーター
- デリゲートのみするメソッド
- 1つのユース ケースの「フレームワーク」

**なぜ悪いのか：**
- 利益なしで複雑さ
- YAGNI（あなたはそれが必要ないでしょう）
- 理解するのが難しい

**リファクタリング：**
- 階層を折りたたむ
- クラスをインライン化
- パラメーターを削除
- メソッドの名前を変更

## カップラー

クラス間の過度なカップリングを表すスメル。

### Feature Envy

**サイン：**
- メソッドは独自よりも別のクラスのデータを使用
- 別のオブジェクトへの多くのゲッター呼び出し
- データと動作は分離

**なぜ悪いのか：**
- 動作の間違った場所
- 不十分なカプセル化
- 保守するのが難しい

**リファクタリング：**
- メソッドを移動
- フィールドを移動
- メソッドを抽出（その後移動）

**例（前）：**
```javascript
class Order {
  getDiscountedPrice(customer) {
    // 顧客データを重く使用
    if (customer.loyaltyYears > 5) {
      return this.price * customer.discountRate;
    }
    return this.price;
  }
}
```

**例（後）：**
```javascript
class Customer {
  getDiscountedPriceFor(price) {
    if (this.loyaltyYears > 5) {
      return price * this.discountRate;
    }
    return price;
  }
}
```

### Inappropriate Intimacy

**サイン：**
- クラスは互いのプライベート パーツにアクセス
- 双方向参照
- サブクラスはパレンツについてあまり知っています

**なぜ悪いのか：**
- 高いカップリング
- 変更はカスケード
- 1つを変更せずに変更するのは難しい

**リファクタリング：**
- メソッドを移動
- フィールドを移動
- 双方向を単方向に変更
- クラスを抽出
- デリゲートを非表示にする

---

### Message Chains

**サイン：**
- 長いメソッド呼び出しチェーン：`a.getB().getC().getD().getValue()`
- クライアントはナビゲーション構造に依存
- 「列車の難破船」コード

**なぜ悪いのか：**
- 脆い — 任意の変更がチェーンを壊す
- Demeterの法則に違反
- 構造にカップリング

**リファクタリング：**
- デリゲートを非表示にする
- メソッドを抽出
- メソッドを移動

**例：**
```javascript
// 悪い：メッセージ チェーン
const managerName = employee.getDepartment().getManager().getName();

// 良い：デリゲーション を非表示
const managerName = employee.getManagerName();
```

---

### Middle Man

**サイン：**
- クラスは別にデリゲートするだけ
- 半分のメソッドはデリゲーション
- 追加値なし

**なぜ悪いのか：**
- 不要な間接参照
- メンテナンス オーバーヘッド
- 混乱したアーキテクチャ

**リファクタリング：**
- 中間の人を削除
- メソッドをインライン化

## スメル重大度ガイド

| 重大度 | 説明 | アクション |
|----------|-------------|--------|
| **Critical** | 開発をブロック、バグを引き起こす | すぐに修正 |
| **High** | 重大なメンテナンス上の負担 | 現在のスプリントで修正 |
| **Medium** | 目立つが管理可能 | 近い将来の計画 |
| **Low** | 軽微な不便 | 機会的に修正 |
