# コード リファクタリング スキル

Martin Fowlerの *Refactoring: Improving the Design of Existing Code*（第2版）に基づいた体系的なアプローチ。このスキルは、テストに支援された安全で増分的な変更を強調しています。

> 「リファクタリングは、コードの外部動作を変更せずにソフトウェア システムを変更するプロセスです。」— Martin Fowler

## コア原則

1. **動作保存**：外部動作は変更されたままである必要があります
2. **小さなステップ**：小さく、テスト可能な変更を行う
3. **テスト駆動**：テストは安全ネット
4. **継続的**：リファクタリングは1回限りのイベントではなく、進行中です
5. **協調的**：各段階でユーザーの承認が必要

## ワークフロー概要

```
フェーズ1：研究と分析
    ↓
フェーズ2：テスト カバレッジ評価
    ↓
フェーズ3：コードの匂い識別
    ↓
フェーズ4：リファクタリング計画作成
    ↓
フェーズ5：増分実装
    ↓
フェーズ6：レビューとイテレーション
```

---

## フェーズ1：研究と分析

### 目的
- コードベース構造と目的を理解する
- リファクタリングの範囲を特定する
- ビジネス要件に関する情報を収集する

### ユーザーに尋ねる質問
始める前に、以下を明確にします：

1. **範囲**: リファクタリングが必要なファイル/モジュール/関数？
2. **目標**: 解決しようとしている問題は何ですか？（可読性、パフォーマンス、保守性）
3. **制約**: 変更してはいけない領域はありますか？
4. **タイムラインのプレッシャー**: これは他の作業をブロックしていますか？
5. **テスト ステータス**: テストが存在しますか？合格していますか？

### アクション
- [ ] ターゲット コードを読んで理解する
- [ ] 依存関係と統合を特定する
- [ ] 現在のアーキテクチャをドキュメント化する
- [ ] 既存の技術的負債マーカー（TODO、FIXME）をメモする

### 出力
ユーザーへの調査結果を提示：
- コード構造のサマリー
- 特定された問題領域
- 初期推奨事項
- **進める前に承認をリクエスト**

## フェーズ2：テスト カバレッジ評価

### テストが重要な理由
> 「テストがないリファクタリングは、シートベルトなしで運転するのに似ています。」— Martin Fowler

テストは**安全なリファクタリングの主要な実現要因**です。テストがなければ、バグが発生するリスクがあります。

### 評価ステップ

1. **既存テストを確認**
   ```bash
   # テスト ファイルを探す
   find . -name "*test*" -o -name "*spec*" | head -20
   ```

2. **既存テストを実行**
   ```bash
   # JavaScript/TypeScript
   npm test

   # Python
   pytest -v

   # Java
   mvn test
   ```

3. **カバレッジを確認（利用可能な場合）**
   ```bash
   # JavaScript
   npm run test:coverage

   # Python
   pytest --cov=.
   ```

### 決定ポイント：ユーザーに質問する

**テストが存在して合格する場合：**
- フェーズ3に進む

**テストが不足または不完全な場合：**
オプションを提示：
1. 最初にテストを作成する（推奨）
2. リファクタリング中に増分的にテストを追加
3. テストなしで進める（危険 - ユーザーの確認が必要）

**テストが失敗する場合：**
- 停止。リファクタリング前に失敗するテストを修正
- ユーザーに尋ねます：最初にテストを修正する必要がありますか？

### テスト作成ガイドライン（必要な場合）

リファクタリングされている各関数について、テストがカバーしていることを確認します：
- 満足な経路（通常の操作）
- エッジ ケース（空の入力、null、境界）
- エラー シナリオ（無効な入力、例外）

「赤-緑-リファクタリング」サイクルを使用：
1. 失敗するテストを作成（赤）
2. パスさせる（緑）
3. リファクタリング

## フェーズ3：コード スメルの識別

### コード スメルとは何ですか？
コード内のより深い問題の症状。バグではありませんが、コードが改善される可能性があることを示す指標です。

### チェックすべき一般的なコード スメル

完全なカタログについては、[references/code-smells.md](references/code-smells.md) を参照してください。

#### クイック リファレンス

| スメル | サイン | 影響 |
|-------|-------|--------|
| **Long Method** | メソッド > 30-50行 | 理解、テスト、保守が難しい |
| **Duplicated Code** | 複数の場所で同じロジック | 複数の場所でのバグ修正が必要 |
| **Large Class** | 責任が多すぎるクラス | 単一責任原則に違反 |
| **Feature Envy** | メソッドは別のクラスのデータを使用 | 不十分なカプセル化 |
| **Primitive Obsession** | オブジェクトの代わりにプリミティブを過度に使用 | ドメイン概念が不足 |
| **Long Parameter List** | メソッドに4以上のパラメーター | 正しく呼び出すのが難しい |
| **Data Clumps** | 同じデータ項目が一緒に表示される | 抽象化が不足 |
| **Switch Statements** | 複雑なswitch/if-elseチェーン | 拡張が難しい |
| **Speculative Generality** | 「万が一」のコード | 不要な複雑さ |
| **Dead Code** | 使用されていないコード | 混乱、保守上の負担 |

### 分析ステップ

1. **自動分析**（スクリプトが利用可能な場合）
   ```bash
   python scripts/detect-smells.py <file>
   ```

2. **手動レビュー**
   - コード全体を体系的に確認
   - 場所と重大度とともに各スメルをメモ
   - Critical/High/Medium/Lowで分類

3. **優先順位付け**
   以下に影響を与えるスメルに焦点を当てる：
   - 現在の開発をブロック
   - バグや混乱を引き起こす
   - 最も変更されたコード パスに影響

### 出力：スメル レポート

ユーザーに提示：
- 位置に特定されたスメルのリスト
- 各スメルの重大度評価
- 推奨される優先度順序
- **優先度に対して承認をリクエスト**

## フェーズ4：リファクタリング計画作成

### リファクタリング選択

各スメルについて、カタログから適切なリファクタリングを選択します。

完全なリストについては、[references/refactoring-catalog.md](references/refactoring-catalog.md) を参照してください。

#### スメル対リファクタリング マッピング

| コード スメル | 推奨リファクタリング |
|------------|---------------------------|
| Long Method | 抽出メソッド、Tempをクエリで置き換える |
| Duplicated Code | 抽出メソッド、メソッドをプルアップ、テンプレート メソッドを形成 |
| Large Class | クラスを抽出、サブクラスを抽出 |
| Feature Envy | メソッドを移動、フィールドを移動 |
| Primitive Obsession | プリミティブをオブジェクトで置き換える、型コードをクラスで置き換える |
| Long Parameter List | パラメーター オブジェクトを導入、オブジェクト全体を保持 |
| Data Clumps | クラスを抽出、パラメーター オブジェクトを導入 |
| Switch Statements | 条件を多態性で置き換える |
| Speculative Generality | 階層を折りたたむ、クラスをインライン化、デッド コードを削除 |
| Dead Code | デッド コードを削除 |

### 計画構造

[templates/refactoring-plan.md](templates/refactoring-plan.md) のテンプレートを使用します。

リファクタリングごと：
1. **ターゲット**: どのコードが変更されますか
2. **スメル**: どの問題に対応しますか
3. **リファクタリング**: どのテクニックを適用するか
4. **ステップ**：詳細なマイクロステップ
5. **リスク**: 何が起こりうるか
6. **ロールバック**: 必要に応じて元に戻す方法

### 段階的なアプローチ

**重要**：段階的にリファクタリングを導入します。

**フェーズA：クイック ウィン**（低リスク、高価値）
- 変数を明確さのために名前変更
- 明白な重複コードを抽出
- デッド コードを削除

**フェーズB：構造の改善**（中程度のリスク）
- 長い関数からメソッドを抽出
- パラメーター オブジェクトを導入
- メソッドを適切なクラスに移動

**フェーズC：アーキテクチャ上の変更**（より高いリスク）
- 条件を多態性で置き換える
- クラスを抽出
- デザイン パターンを導入

### 決定ポイント：ユーザーに計画を提示する

実装する前：
- 完全なリファクタリング計画を表示
- 各フェーズとそのリスクを説明
- 各フェーズの明示的な承認を取得
- **「フェーズAを進めるべきですか？」と尋ねる**

## フェーズ5：増分実装

### 黄金的ルール
> 「変更 → テスト → グリーン？ → コミット → 次のステップ」

### 実装リズム

リファクタリング ステップごと：

1. **事前チェック**
   - テストが合格している（グリーン）
   - コード コンパイル

2. **1つの小さな変更を加える**
   - カタログのメカニクスに従う
   - 変更を最小限に保つ

3. **検証**
   - テストをすぐに実行
   - コンパイル エラーをチェック

4. **テストが合格する場合（グリーン）**
   - 説明的なメッセージでコミット
   - 次のステップに進む

5. **テストが失敗する場合（赤）**
   - すぐに停止
   - 変更を元に戻す
   - 何が悪かったのかを分析
   - 不明な場合はユーザーに問い合わせ

### コミット戦略

各コミットは：
- **アトミック**：1つの論理的変更
- **可逆的**：簡単に戻す
- **説明的**：明確なコミット メッセージ

コミット メッセージの例：
```
refactor: processOrder()からcalculateTotal()を抽出
refactor: 明確さのために「x」を「customerCount」に名前変更
refactor: 未使用のvalidateOldFormat()メソッドを削除
```

### 進行状況のレポート

各サブフェーズの後、ユーザーにレポート：
- 行われた変更
- テストはまだ合格していますか？
- 遭遇した問題
- **「次のバッチで続ける？」と尋ねる**

## フェーズ6：レビューとイテレーション

### リファクタリング後のチェックリスト

- [ ] すべてのテストが合格している
- [ ] 新しい警告/エラーなし
- [ ] コード コンパイル成功
- [ ] 動作が変更されていない（手動検証）
- [ ] ドキュメンテーションが必要な場合は更新
- [ ] コミット履歴がクリーン

### メトリクス比較

リファクタリング前後に複雑度分析を実行：
```bash
python scripts/analyze-complexity.py <file>
```

改善を提示：
- コード行数の変更
- 循環的複雑度の変化
- 保守性インデックスの変化

### ユーザー レビュー

最終結果を提示：
- すべての変更のサマリー
- コード比較の前後
- メトリクスの改善
- 残存する技術的負債
- **「これらの変更に満足していますか？」と尋ねる**

### 次のステップ

ユーザーと議論：
- 対処する追加のスメル？
- フォローアップ リファクタリングをスケジュール
- 他の場所でも同様の変更を適用する？

## 重要なガイドライン

### 停止して質問する場合

常に一時停止してユーザーに相談する場合：
- ビジネス ロジックについて不確実
- 変更が外部APIに影響する可能性がある
- テスト カバレッジが不十分
- 重要なアーキテクチャの決定が必要
- リスク レベルが増加する
- 予期しない複雑さが発生する

### セーフティ ルール

1. **テストなしでリファクタリングしない**（ユーザーが明示的にリスクを認めない限り）
2. **大きな変更を行わない** - 小さなステップに分割
3. **テスト実行をスキップしない** 各変更後
4. **テストが失敗しても続行しない** - 最初に修正またはロールバック
5. **不確定な場合は尋ねる** - 当然

### しないこと

- リファクタリングと機能追加を組み合わせない
- 本番環境の緊急事態中にリファクタリングしない
- 理解していないコードをリファクタリングしない
- 過度なエンジニアリング - シンプルに保つ
- 一度にすべてをリファクタリングしない

## クイック スタート例

### シナリオ：重複を含むロング メソッド

**前：**
```javascript
function processOrder(order) {
  // 150行のコード：
  // - 重複検証ロジック
  // - インライン計算
  // - 混在した責任
}
```

**リファクタリング ステップ：**

1. **テストが存在することを確認** processOrder()
2. **抽出** validateOrder()に検証
3. **テスト** - 合格する必要があります
4. **抽出** calculateOrderTotal()に計算
5. **テスト** - 合格する必要があります
6. **抽出** notifyCustomer()に通知
7. **テスト** - 合格する必要があります
8. **レビュー** - processOrder()は3つの明確な関数を調整します

**後：**
```javascript
function processOrder(order) {
  validateOrder(order);
  const total = calculateOrderTotal(order);
  notifyCustomer(order, total);
  return { order, total };
}
```
