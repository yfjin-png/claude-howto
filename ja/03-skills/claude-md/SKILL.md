## ユーザー入力

```text
$ARGUMENTS
```

続行する前に、ユーザー入力を**必ず**検討してください（空の場合は除く）。ユーザーが指定する場合があります：
- `create` - 新しいCLAUDE.mdをゼロから作成
- `update` - 既存のCLAUDE.mdを改善
- `audit` - 現在のCLAUDE.mdの品質を分析してレポート
- 作成/更新する特定のパス（例：`src/api/CLAUDE.md` ディレクトリ固有の指示）

## コア原則

**LLMはステートレス**: CLAUDE.mdはすべての会話に自動的に含まれる唯一のファイルです。AIエージェント用のコードベースへの主要なオンボーディング ドキュメントとして機能します。

### 黄金のルール

1. **簡潔性優先**: フロンティア LLMは約150～200の指示に従うことができます。Claude Codeのシステム プロンプトは既に約50を使用しています。CLAUDE.mdを焦点を絞った簡潔なものに保ちます。

2. **普遍的な適用性**: すべてのセッションに関連する情報のみを含めます。タスク固有の指示は別ファイルに属します。

3. **Claudeをリンターとして使用しないでください**: スタイル ガイドラインはコンテキストを膨らませ、指示を遵守する機能を低下させます。代わりに決定論的なツール（prettier、eslint など）を使用してください。

4. **自動生成しないでください**: CLAUDE.mdはAIハーネスの最高レベルのポイントです。注意深く検討して手動で作成します。

## 実行フロー

### 1. プロジェクト分析

最初に現在のプロジェクト状態を分析します：

1. 既存のCLAUDE.mdファイルを確認：
   - ルート レベル：`./CLAUDE.md` または `.claude/CLAUDE.md`
   - ディレクトリ固有：`**/CLAUDE.md`
   - グローバル ユーザー設定：`~/.claude/CLAUDE.md`

2. プロジェクト構造を特定：
   - テック スタック（言語、フレームワーク）
   - プロジェクト タイプ（モノレポ、単一アプリ、ライブラリ）
   - 開発ツール（パッケージ マネージャー、ビルド システム、テスト ランナー）

3. 既存のドキュメンテーションを確認：
   - README.md
   - CONTRIBUTING.md
   - package.json、pyproject.toml、Cargo.toml など

### 2. コンテンツ戦略（WHAT、WHY、HOW）

3つの次元でCLAUDE.mdを構成します：

#### WHAT - テクノロジーと構造
- テック スタック概要
- プロジェクト組織（モノレポにとって特に重要）
- 主要なディレクトリとその目的

#### WHY - 目的とコンテキスト
- プロジェクトが何をするのか
- 特定のアーキテクチャ上の決定が行われた理由
- 各主要なコンポーネントが何を担当するか

#### HOW - ワークフローと規則
- 開発ワークフロー（bun対node、pip対uvなど）
- テスト手順とコマンド
- 検証とビルド方法
- 重要な「落とし穴」または非明白な要件

### 3. プログレッシブ ディスクロージャー戦略

大規模なプロジェクトの場合、`agent_docs/` フォルダを作成することを推奨します：

```
agent_docs/
  |- building_the_project.md
  |- running_tests.md
  |- code_conventions.md
  |- architecture_decisions.md
```

CLAUDE.mdで、これらのファイルを次のような指示で参照します：
```markdown
詳細なビルド指示については、`agent_docs/building_the_project.md` を参照してください
```

**重要**: コード スニペットの代わりに、`file:line` 参照を使用して、古いコンテキストを避けてください。

### 4. 品質制約

CLAUDE.mdを作成または更新する場合：

1. **ターゲット長**: 300行以下（理想的には100行以下）
2. **スタイルルールなし**: リントやフォーマット指示を削除
3. **タスク固有の指示なし**: 別ファイルに移動
4. **コード スニペットなし**: ファイル参照を使用
5. **冗長情報なし**: package.jsonまたはREADMEで何が繰り返されるかは繰り返さないでください

### 5. 重要なセクション

よく構成されたCLAUDE.mdに含めるべきもの：

```markdown
# プロジェクト名

簡潔な1行の説明。

## テック スタック
- 主言語とバージョン
- 主なフレームワーク/ライブラリ
- データベース/ストレージ（存在する場合）

## プロジェクト構造
[モノレポまたは複雑な構造のみ]
- `apps/` - アプリケーション エントリ ポイント
- `packages/` - 共有ライブラリ

## 開発コマンド
- インストール：`command`
- テスト：`command`
- ビルド：`command`

## 重要な規則
[非明白で高影響力の規則のみ]
- 簡潔な説明を含む規則1
- 簡潔な説明を含む規則2

## 既知の問題/落とし穴
[開発者を困らせる傾向がある事項]
- 問題1
- 問題2
```

### 6. 避けるべきアンチパターン

**含めないでください：**
- コード スタイル ガイドライン（リンターを使用）
- Claudeの使用方法に関するドキュメンテーション
- 明白なパターンの長い説明
- コピーして貼り付けたコード例
- 一般的なベストプラクティス（「クリーン コードを書く」）
- 特定のタスク用の指示
- 自動生成されたコンテンツ
- 広範なTODOリスト

### 7. 検証チェックリスト

ファイナライズする前に確認：

- [ ] 300行以下（理想的には100行以下）
- [ ] すべての行がすべてのセッションに適用される
- [ ] スタイル/フォーマット ルールなし
- [ ] コード スニペットなし（ファイル参照を使用）
- [ ] コマンドは動作確認済み
- [ ] 複雑なプロジェクトに使用されるプログレッシブ ディスクロージャー
- [ ] 重要な落とし穴はドキュメント化される
- [ ] README.mdとの冗長性なし

## 出力フォーマット

### `create` または デフォルト：

1. プロジェクトを分析
2. 上記の構造に従ってCLAUDE.mdドラフトを作成
3. レビュー用にドラフトを提示
4. 承認後、適切な場所に書き込み

### `update`:

1. 既存のCLAUDE.mdを読む
2. ベストプラクティスに対して監査
3. 特定します：
   - 削除するコンテンツ（スタイル ルール、コード スニペット、タスク固有）
   - 凝縮するコンテンツ
   - 不足している重要な情報
4. レビュー用に変更を提示
5. 承認後に変更を適用

### `audit`:

1. 既存のCLAUDE.mdを読む
2. 生成レポート：
   - 現在の行数対ターゲット
   - 普遍的に適用可能なコンテンツの割合
   - 見つかったアンチパターンのリスト
   - 改善の推奨事項
3. ファイルを変更しない、レポートのみ

## AGENTS.md処理

ユーザーがAGENTS.md作成/更新をリクエストする場合：

AGENTS.mdは特殊化されたエージェント動作を定義するために使用されます。CLAUDE.md（プロジェクト コンテキスト用）とは異なり、AGENTS.mdは以下を定義します：
- カスタム エージェント ロールと機能
- エージェント固有の指示と制約
- マルチエージェント シナリオのワークフロー定義

同様の原則を適用します：
- 焦点を絞った簡潔な状態に保つ
- プログレッシブ ディスクロージャーを使用
- コンテンツを埋め込む代わりに外部ドキュメントを参照

## ノート

- 含める前にコマンドが動作することを確認
- 疑わしい場合は、除外してください - 簡潔性優先
- システムの通知はCLAUDE.mdが「関連する場合と関連しない場合がある」ことを示します。ノイズが多いほど、無視されやすくなります
- モノレポはWHAT/WHY/HOW構造から最も利益を得る
- ディレクトリ固有のCLAUDE.mdファイルはさらに焦点を絞った状態にする必要があります
