## ユーザー入力

```text
$ARGUMENTS
```

続行する前に、ユーザー入力を**必ず**検討してください。ユーザーは以下を提供する必要があります：
- **アイデア/トピック**: ブログ投稿の主要な概念またはテーマ
- **リソース**: URL、ファイル、または研究への参照（オプションですが推奨）
- **ターゲット オーディエンス**: ブログ投稿の対象（オプション）
- **トーン/スタイル**: 公式、カジュアル、技術的など（オプション）

**重要**: ユーザーが**既存のブログ投稿**の更新をリクエストしている場合、ステップ0～8をスキップして、直接**ステップ9**から開始します。既存のドラフト ファイルを最初に読んで、次にイテレーション プロセスを実行します。

## 実行フロー

これらのステップに従ってください。指定された箇所でユーザーの承認なしに、ステップをスキップまたは進めないでください。

### ステップ0：プロジェクト フォルダを作成する

1. `YYYY-MM-DD-short-topic-name` 形式を使用してフォルダ名を生成します
   - 今日の日付を使用します
   - トピックから短いURLフレンドリーなスラッグを作成します（小文字、ハイフン、最大5語）

2. フォルダ構造を作成します：
   ```
   blog-posts/
   └── YYYY-MM-DD-short-topic-name/
       └── resources/
   ```

3. 進める前にフォルダ作成をユーザーと確認します。

### ステップ1：研究とリソース収集

1. ブログ投稿ディレクトリに `resources/` サブフォルダを作成します

2. 提供された各リソースについて：
   - **URL**: フェッチしてキー情報を `resources/` にマークダウン ファイルとして保存します
   - **ファイル**: 読み込んで `resources/` で要約します
   - **トピック**: Webから最新情報を収集するには検索を使用します

3. 各リソースについて、`resources/` にサマリー ファイルを作成します：
   - `resources/source-1-[short-name].md`
   - `resources/source-2-[short-name].md`
   - など

4. 各サマリーに含める必要があります：
   ```markdown
   # ソース：[タイトル/URL]

   ## 主なポイント
   - ポイント1
   - ポイント2

   ## 関連する引用/データ
   - 引用またはスタッツ1
   - 引用またはスタッツ2

   ## これがトピックとどのように関連するか
   関連性の簡潔な説明
   ```

5. ユーザーに研究サマリーを提示します。

### ステップ2：ブレインストーミングと明確化

1. アイデアと調査されたリソースに基づいて、以下を提示します：
   - **主なテーマ** 研究から特定
   - **ブログ投稿の可能性のある角度**
   - **カバーする必要があるキー ポイント**
   - **明確化が必要な情報のギャップ**

2. 明確化の質問をします：
   - 読者に持ってほしい主なポイントは何ですか？
   - 研究から強調したい特定のポイントはありますか？
   - ターゲット長はどのくらいですか？（短：500～800語、中：1000～1500、長：2000以上）
   - 除外したいポイントはありますか？

3. **ユーザーの応答を待ってから進めます。**

### ステップ3：アウトラインを提案する

1. 以下を含む構造化アウトラインを作成します：

   ```markdown
   # ブログ投稿アウトライン：[タイトル]

   ## メタ情報
   - **ターゲット オーディエンス**：[誰]
   - **トーン**：[スタイル]
   - **ターゲット長**：[単語数]
   - **主なポイント**：[キー メッセージ]

   ## 提案される構造

   ### フック/イントロダクション
   - オープニング フック アイデア
   - コンテキスト設定
   - テーゼステートメント

   ### セクション1：[タイトル]
   - キー ポイントA
   - キー ポイントB
   - [ソース]からの証拠をサポート

   ### セクション2：[タイトル]
   - キー ポイントA
   - キー ポイントB

   [すべてのセクションに対して続行...]

   ### まとめ
   - キー ポイントのサマリー
   - 行動喚起または最後の考え

   ## 引用するソース
   - ソース1
   - ソース2
   ```

2. ユーザーにアウトラインを提示して、**承認または変更を依頼します。**

### ステップ4：承認されたアウトラインを保存する

1. ユーザーがアウトラインを承認したら、ブログ投稿フォルダに `OUTLINE.md` として保存します。

2. アウトラインが保存されたことを確認します。

### ステップ5：アウトラインをコミット（gitリポジトリの場合）

1. 現在のディレクトリがgitリポジトリかどうかを確認します。

2. はい場合：
   - 新しいファイルをステージします：ブログ投稿フォルダ、リソース、OUTLINE.md
   - コミット メッセージで作成します：`docs: Add outline for blog post - [topic-name]`
   - リモートにプッシュ

3. gitリポジトリでない場合、このステップをスキップしてユーザーに通知します。

### ステップ6：ドラフトを書く

1. 承認されたアウトラインに基づいて、完全なブログ投稿ドラフトを書きます。

2. OUTLINE.mdの構造に正確に従います。

3. 以下を含めます：
   - フックのある魅力的なイントロダクション
   - 明確なセクション ヘッダー
   - 研究からのサポート証拠と例
   - セクション間のスムーズなトランジション
   - ポイントを押さえた強い結論
   - **引用**：すべての比較、統計、データ ポイント、事実上の請求は、元のソースを引用**する必要があります**

4. ドラフトを `draft-v0.1.md` としてブログ投稿フォルダに保存します。

5. フォーマット：
   ```markdown
   # [ブログ投稿タイトル]

   *[オプション：サブタイトルまたはキャッチフレーズ]*

   [インライン引用を含む完全なコンテンツ...]

   ---

   ## リファレンス
   - [1] ソース1タイトル - URLまたは引用
   - [2] ソース2タイトル - URLまたは引用
   - [3] ソース3タイトル - URLまたは引用
   ```

6. **引用要件**：
   - すべてのデータ ポイント、統計、比較**を引用する必要があります**
   - 番号付き参照[1]、[2]などを使用するか、名前付き引用[ソース名]を使用します
   - セクションの末尾の参照セクションへのリンク引用
   - 例：「開発者の65%がTypeScriptを好む研究[1]」
   - 例：「Reactはレンダリング速度でVueを20%上回ります[React Benchmarks 2024]」

### ステップ7：ドラフトをコミット（gitリポジトリの場合）

1. gitリポジトリにあるかどうかを確認します。

2. はい場合：
   - ドラフト ファイルをステージします
   - コミット メッセージで作成します：`docs: Add draft v0.1 for blog post - [topic-name]`
   - リモートにプッシュ

3. gitリポジトリでない場合、スキップしてユーザーに通知します。

### ステップ8：レビュー用にドラフトを提示する

1. ドラフト コンテンツをユーザーに提示します。

2. フィードバックを要求します：
   - 全体的な印象？
   - 拡張または削減が必要なセクション？
   - トーン調整が必要ですか？
   - 不足している情報？
   - 特定の編集または書き直し？

3. **ユーザーの応答を待ちます。**

### ステップ9：イテレートまたはファイナライズ

**ユーザーが変更をリクエストする場合：**
1. すべてのリクエストされた変更をメモします
2. 次の調整でステップ6に戻ります：
   - バージョン番号をインクリメント（v0.2、v0.3など）
   - すべてのフィードバックを組み込む
   - `draft-v[X.Y].md` として保存します
   - ステップ7～8を繰り返す

**ユーザーが承認する場合：**
1. 最終ドラフト バージョンを確認します
2. ユーザーがリクエストする場合は、必要に応じて `final.md` に名前変更します
3. ブログ投稿作成プロセスをサマリーします：
   - 作成されたバージョン数
   - バージョン間の主な変更
   - 最終単語数
   - 作成されたファイル

## バージョン追跡

すべてのドラフトは段階的なバージョン管理で保存されます：
- `draft-v0.1.md` - 初期ドラフト
- `draft-v0.2.md` - 最初のフィードバック ラウンド後
- `draft-v0.3.md` - 2番目のフィードバック ラウンド後
- など

これにより、ブログ投稿の進化を追跡し、必要に応じて戻すことができます。

## 出力ファイル構造

```
blog-posts/
└── YYYY-MM-DD-topic-name/
    ├── resources/
    │   ├── source-1-name.md
    │   ├── source-2-name.md
    │   └── ...
    ├── OUTLINE.md
    ├── draft-v0.1.md
    ├── draft-v0.2.md (イテレーションの場合)
    └── draft-v0.3.md (より多くのイテレーションの場合)
```

## 品質のためのヒント

- **フック**: 質問、驚きの事実、または関連する情報から始めます
- **フロー**: 各段落は次に接続する必要があります
- **証拠**: 研究からの主張をデータでサポート
- **引用**：**常に**以下のソースを引用します：
  - すべての統計とデータ ポイント（例：「ソースによると、開発者の75%が...」）
  - 製品、サービス、またはアプローチ間の比較（例：「Xはソース]よりYより2倍高速です」）
  - 市場トレンド、研究成果物、ベンチマークに関する事実上の請求
  - インライン引用形式を使用：[ソース名]または[著者、年]
- **ボイス**: トーン全体を通して一貫した音声を維持
- **長さ**: ターゲット単語数に従う
- **可読性**: 短い段落、適切な箇所での箇条書きを使用
- **CTA**: 明確な行動喚起または考えさせる質問で終わる

## ノート

- 概要されたチェックポイントでユーザーの承認を常に待つ
- すべてのドラフト バージョンを履歴用に保持
- URLが提供されるときの最新情報用にWebを検索する
- リソースが不十分な場合、より多くを要求するか、追加の研究を提案
- ターゲット オーディエンス（技術的、一般的、ビジネスなど）に基づいてトーンを調整
